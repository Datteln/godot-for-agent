# 结构化历史会话块存储与渲染 — 需求文档

## 一、现状与问题

### 1.1 当前数据流

```
Session JSON (frame messages)
    │
    ├── _history_items_for_frame()     ──┐
    │   → SessionHistoryItemDTO(role, text)  │  全部扁平化为文本
    │                                    │
    └── EventStore (in-memory)           │
        → _history_items_for_events()  ──┘
            → [事件块整体追加到帧消息之后]

        ↓
    SessionHistoryResponse.items[]
        ↓
    前端 _handle_session_history()
        → 用文本前缀猜测块类型（begins_with "Thought for"/"Read "/"Edit "...）
        → 调用对应渲染函数
```

### 1.2 核心问题清单

| # | 问题 | 根因 |
|---|------|------|
| P1 | Thought 用时固定显示 `0.00s` | `agent_reasoning_delta` 仅在内存 EventStore，服务重启后丢失；fallback 硬编码 `0.00s` |
| P2 | Plan/Step/Verify 事件显示在所有对话末尾，而非插在对应工具调用处 | `_history_items_for_events()` 整体追加，未按时间穿插到帧消息中 |
| P3 | Grep 条目历史只有首行（无匹配结果列表） | `_format_tool_result_summary` 只生成 `Grep "x" (in project)` 一行文本，Detail 数据丢失 |
| P4 | Plan 步骤列表在历史中格式错乱 | `_format_create_plan_history_summary` 生成文本后由 `append_workflow_summary_entry` 二次解析，两端格式约定不统一 |
| P5 | 助手回复中间文本（`Thought: ...`）未渲染为折叠 `✻` 条目 | 前端 `_apply_thought_prefix` + `_append_message` 路径未触发 `_append_history_thought_item` |
| P6 | `_handle_session_history()` 文本猜测逻辑脆弱 | 依赖文本前缀规则，思考正文中若有 `"Read "` 开头行会被误切成 Read 条目 |
| P7 | 旧格式 guard（`get_child_count() > 0`）在事件提前写入消息时静默放弃整个历史加载 | 已在上次修复中替换为 session_id 校验，但根源（文本模型）未解 |

---

## 二、目标

1. 后端 `GET /sessions/{id}/history` 返回**结构化块数组** `blocks[]`，每块携带 `type` 及类型专属字段，不再是扁平 `{role, text}`
2. Plan/Step/Verify/Reasoning 等事件与帧消息**按时间顺序穿插**，不再整体后置
3. 前端 `_handle_session_history()` 按 `type` 直接分发到对应渲染函数，**零文本解析**
4. 历史加载效果与实时对话视觉完全一致（折叠 Thought、Grep 可展开结果、Edit 行统计等）
5. 兼容旧格式（无 `blocks` 字段时降级到现有文本路径）

### 3.4 不持久化的块

| 块 | 处理方式 |
|----|---------|
| ⑩ 实时 Reasoning | 流结束后转为 `ThoughtBlock`（在 `_emit` 时记录 `elapsed_ms`） |
| ⑪ 实时流式文本 | `final` 处理后转为对应类型块（`log_text` / `log_edit` / `log_read`） |
| ⑫ Tool Diff 面板 | 历史中降级为 `LogEditBlock`（保留路径和行统计，不保存 before/after 内容） |
| ⑬ 内联确认框 | 以工具结果块替代，不单独持久化 |
| ⑭ 待处理通知 | 由 `pending_turn_id != null` 状态驱动，不是内容块 |

## 四、后端改造

### 4.1 `session_history()` 重写逻辑

**现在**：先遍历所有帧生成 items，再将所有事件追加到末尾。  
**新**：对每个帧，在处理 assistant/tool message 的同时，按 `(frame_id, loop, seq)` 查找相邻事件，按时间顺序穿插插入。

```
for frame in session.agent_stack:
    for message in frame.messages:
        blocks += _message_to_blocks(message, frame)
        # 同时查找该帧该轮次之前未处理的相邻事件
        adjacent_events = event_index.pop_events_before(frame.id, message_seq)
        blocks += [_event_to_block(e) for e in adjacent_events]

# 追加剩余事件（服务端工具结果、最终 reasoning delta 等）
blocks += [_event_to_block(e) for e in event_index.remaining()]
```

事件索引以 `(frame_id, loop)` 为 key，用于将 `plan_step_started` 等与对应工具调用时间对齐。

### 4.2 帧消息 → Block 映射表

| frame message 形态 | 生成 Block |
|-------------------|-----------|
| `role=user` | `UserBlock{text}` |
| `role=assistant, content="Thought: X\n\nbody"` | `ThoughtBlock{header="Thought: X", detail=body}` |
| `role=assistant, content="正文"（无工具调用）` | `LogTextBlock{text, indent=true, marker=false}` |
| `role=assistant, tool_calls` | 忽略（宣告不显示） |
| `role=tool, tool ∈ READ_TOOLS` | `LogReadBlock{path, line_start=1, line_end=N}` |
| `role=tool, tool ∈ EDIT_TOOLS` | `LogEditBlock{path, added, removed}` |
| `role=tool, tool ∈ GREP_TOOLS` | `LogGrepBlock{pattern, include, match_count, results[], truncated}` |
| `role=tool, tool=create_plan` | `PlanCreatedBlock{summary, steps[]}` |
| `role=tool, tool=delegate_many` | `DelegateResultsBlock{results[]}` |
| `role=tool, tool=delegate` | `DelegateResultBlock{agent, summary}` |
| `role=tool` 其他（如 run_tests） | `LogTextBlock{text=摘要, marker=true}` |
| `role=system`（verify JSON / compact_summary） | 忽略（由事件块负责） |

| event.type | 生成 Block |
|-----------|-----------|
| `agent_reasoning_delta` | `ThoughtBlock{header="Thought for {elapsed:.2f}s", detail=text}`（仅取最后一个 delta） |

| `user_submitted` | `SystemTextBlock{text="消息已提交"}` |
| `server_tool_result` kind=read | `LogReadBlock{...}` |

### 4.4 Reasoning 用时恢复

在 `orchestrator/agent.py` 的 `_emit("agent_reasoning_delta", payload)` 中，额外附加：
```python
payload["elapsed_ms"] = int((time.monotonic() - reasoning_started_at) * 1000)
```

`_history_item_for_stream_event()` 读取 `elapsed_ms` 生成：
```python
header = f"Thought for {elapsed_ms / 1000:.2f}s"
```

服务重启后事件丢失，`ThoughtBlock` 来自帧消息的 `"Thought: ..."` 前缀，`header` 显示为 `"Thought"` 无用时（不显示 0.00s，避免误导）。

## 五、前端改造

### 5.1 `_handle_session_history()` 主流程

```gdscript
func _handle_session_history(response: Dictionary) -> void:
    # ... 状态检查、session_id 校验、_clear_messages() 不变 ...
    var saved_auto_scroll := _auto_scroll
    _auto_scroll = false
    
    if response.has("blocks"):
        for block in response.get("blocks", []):
            if block is Dictionary:
                _render_history_block(block)
    else:
        # 兼容旧格式：保留现有文本解析路径
        _handle_session_history_legacy(response)
    
    # ... pending_turn_id 处理、history_restored 消息 ... 不变 ...
    _auto_scroll = saved_auto_scroll
    _force_scroll_once = true
    _scroll_to_bottom()
```

### 5.3 需新增的三个辅助渲染函数

**`_render_history_grep_block(block)`**：
从 `results[]` 重建 `"Grep "{pattern}" (in {include})\n{N} match(es)\n{path}:{line} {text}\n..."` 文本，传入 `append_log_stream_message` → `append_grep_entry` 自动处理折叠。

**`_render_history_plan_created(block)`**：
从 `summary` + `steps[]` 重建 `"Plan created:\n{summary}\n\n1. {title} ({agent})\n..."` 文本，传入 `append_log_stream_message` → `append_workflow_summary_entry`。

**`_render_history_delegate_results(block)`**：
从 `results[]` 重建 `"Delegate results:\n\n**1. {agent}**\n{summary}..."` 文本，传入 `append_log_stream_message` → `append_workflow_summary_entry`。
