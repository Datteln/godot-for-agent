## 1. 背景与目标

### 1.1 背景

用 Godot 做游戏时，存在大量**重复性、机械性、模板化**的工作，横跨多个环节：

- **编程**：写样板代码、实现常见功能、解释/重构/修 bug、补全脚本（GDScript 与 C#）。
- **地图**：填充地面、围墙、铺路、批量摆放、生成结构化房间。
- **场景搭建**：创建节点、组装场景树、批量设置属性、实例化预制场景。
- **资源/项目**：批量创建/重命名/整理资源、调整项目设置。

这些工作规则明确、可程序化，却要人工逐个完成，效率低、易出错、消耗创意精力。我们希望在 Godot **编辑器内部**集成一个 AI Agent（基于大语言模型），让开发者用**自然语言**描述意图，由 Agent 调用引擎能力自动完成，把人从重复劳动中解放出来。

### 1.2 项目目标

- **G1（广度）**：Agent 在**编程辅助**与**重复性游戏开发任务**（地图、场景、资源）多个能力域同时提供帮助。
- **G2（自然语言驱动）**：开发者用自然语言下达意图，Agent 通过"工具调用"操作编辑器完成。
- **G3（可控）**：所有改动**先预览、用户确认后才落地**；落地后可经**统一撤销入口**回退（节点/属性走 Godot 原生 UndoRedo，脚本/资源/批量改名等**文件类改动经快照/事务记录回滚**）。
- **G4（准确）**：减少模型幻觉——不写不存在的 API、不绘制不存在的瓦片、不破坏既有工程；代码遵循 GDScript 4.x / 现代 C#。
- **G5（嵌入体验）**：LLM 推理由独立 **Python 服务**承担，随编辑器自动启停，对用户表现为"开箱即用、内嵌引擎"。
- **G6（可扩展）**：能力域与工具集可平滑增长，新增能力不应改动核心协议。
- **G7（开源免费）**：项目以**开源**形式发布；用户**自带大模型 key（BYO key）**，无付费后端、无厂商锁定。功能对标 Ziva（见 §1.5）。

### 1.3 非目标（本期不做）

- 不做游戏运行时（runtime）的 AI 行为/寻路（NavigationAgent 范畴，与本项目无关）。
- 本期不做素材生成；其中**精灵图/3D 模型生成列为后续**（对标 Ziva 的延伸），音频/纹理素材生成不在范围。
- 不做多人协同编辑。
- 不做 3D 网格地图（GridMap）自动建造（列入后续）。
- 不替代开发者的创意与架构决策，仅承担"按指令执行 + 辅助"的部分。

### 1.5 竞品与定位

**现状**：Godot 的 AI 辅助开发工具已形成两大流派——

- **编辑器内插件**（AI 嵌入 Godot 侧边面板）：如 Ziva、AI Assistant Hub、AlphaAgent。
- **MCP 服务器**（外部 AI 客户端如 Claude / Cursor 操控编辑器）：如 godot-mcp、Godot MCP Pro、hi-godot/godot-ai。

**本项目定位**：属于**编辑器内插件**流派，**功能对标 Ziva**，走**完全开源 + 自带 key（BYO key）+ 免费**路线；区别于 Ziva 的**账号额度 / 托管模型服务 + 付费套餐**路线（Ziva 安装器开源，但能力依赖其托管后端与套餐）。本项目强调**完整开源、BYO key、无中间商**。

**对标 Ziva 的功能覆盖**：

| Ziva 核心能力 | 本项目 |
|------|------|
| 编辑器内侧边面板、不切换上下文 | ✅ 覆盖 |
| 写代码 / 加功能 / 修 bug（GDScript + C#） | ✅ 覆盖（M1） |
| 实时操作场景树 | ✅ 覆盖（M1） |
| 智能代码库检索 | ✅ 覆盖（M2，RAG） |
| 多模型支持（ChatGPT / Claude / Gemini） | ✅ 覆盖（用户配端点） |
| 读取调试器 / 运行时错误并修复 | ✅ 补齐（M2） |
| 生成并运行单元测试 | ✅ 补齐（M2） |
| 生成精灵图 / 3D 模型 | ⏳ 列为后续 |

**差异化优势**：开源可审计可定制；用户自带 key、无中间商加价、无厂商锁定。

> **外部核验（2026-06）**：Ziva 官网宣传编辑器内构建场景树、写 GDScript/C#、运行测试、读取调试错误，并提供免费/付费套餐；Godot Asset Library 条目标注其可生成 2D 精灵 / 3D 资产、做基础 QA 测试——其安装器走 MIT/Unlicense，但能力依赖托管模型服务。AI Assistant Hub 则更接近本项目的 BYO / 本地 LLM 桥接路线。来源：ziva.sh、ziva.sh/docs、Godot Asset Library（asset 4879）、AI Assistant Hub（asset 3427）。

### 1.6 创新特性（超越 Ziva 的差异化）

> 以下特性超出 Ziva 当前功能边界，是本项目的差异化招牌。除特别说明外归入**创新阶段（M3，分批落地）**，避免冲击核心范围。改动型特性仍遵循"预览—确认—可撤销"。

| # | 创新特性 | 价值 | 技术依托 | 类型 | 阶段 |
|---|------|------|------|------|------|
| IN-1 | **AI 自动试玩 / 自测** | 闭环"AI 写 → AI 玩 → 报告 → 修"，自动发现卡点/崩溃/平衡问题 | Godot headless 运行 + 脚本注入输入 | 执行 + 只读 | M3（招牌） |
| IN-2 | **草图 / 参考图 → 关卡** | 从"文字描述"升级到"画给它看"，视觉驱动建图 | TileMap API + 多模态模型 | 改动 | M3（招牌） |
| IN-3 | **运行时状态诊断问答** | 暂停时问"为什么敌人不动"，读运行时场景/变量诊断（比读 error 深一层） | 运行时场景树/调试器状态读取 | 只读 | M3 |
| IN-4 | **内容（非美术）生成** | NPC 对话树/剧情/任务链、本地化文本、数值/数据资源批量生成 | LLM 文本能力 + `.tres` 资源 | 改动 | 对话/数据资源可 M2，剧情链 M3 |
| IN-5 | **性能剖析建议** | 接 Profiler 数据，定位瓶颈节点/脚本并给优化方案 | Godot Profiler / 调试器数据 | 只读 | M3 |
| IN-6 | **目标驱动的自主多步执行** | 给高层目标自主规划 → 搭建 → 自测全流程 | Agent 规划 + 多工具编排 + 自测(IN-1) | 改动 | M3（愿景） |
| IN-7 | **看懂现有资产** | 读精灵表自动切帧建动画、看碰撞图配碰撞体、识别 TileSet 配 terrain | 多模态 + 资源 API | 改动 | 精灵表切帧可 M2，其余 M3 |

**定位升级**：

> Ziva 是"在编辑器里帮你写代码的 AI"；本项目要做"**会玩你的游戏、看得懂你的草图、能诊断运行时、能自主搭建**的 Godot AI 助手"。

## 2. 名词与术语

| 术语 | 说明 |
|------|------|
| Agent | 基于 LLM 的智能体，能理解自然语言并通过"工具调用"操作引擎 |
| 能力域（Capability Domain） | Agent 的一类职责，如"编程辅助""地图建造""场景操作" |
| 工具（Tool） | 暴露给 LLM 的原子能力，如"读脚本""写脚本""填充矩形" |
| 工具调用（Tool Calling） | LLM 决定调用某工具并给出参数的机制 |
| 工具三类：只读 / 改动 / 执行 | 只读仅获取信息（直接执行）；改动会修改工程（需预览确认 + 可撤销）；执行会运行游戏/测试（需确认 + 超时 + 取消 + 日志 + 沙箱边界，如 `run_tests`、AI 试玩） |
| 预览—确认落地 | 改动先渲染为预览（diff/操作清单），用户确认后才真正执行 |
| Agent 循环 | "LLM 下指令 → 预览确认 → 执行工具 → 回传结果 → LLM 继续"的多轮往复 |
| 前端 | Godot 编辑器内的插件，提供 UI、预览确认并执行编辑器内工具 |
| LLM 服务 | 独立 Python 进程，调用大模型、维护对话状态、路由工具调用、承担检索/文档接地 |
| ClassDB / 文档接地 | 把 Godot 真实的类与 API 信息喂给模型，降低 API 幻觉 |
| RAG | 检索增强：从工程代码/文档中检索相关片段再喂给模型 |

## 4. 总体架构需求

### 4.1 三层架构（定稿）

```
┌──────────────────────── Godot 编辑器 ────────────────────────┐
│  [前端层] 编辑器插件（EditorPlugin，GDScript @tool）         │
│    · 聊天面板 UI（停靠面板）+ 改动预览/确认面板               │
│    · 收集上下文 + 读 ClassDB 真实签名（节点/脚本/场景树等）  │
│    · 执行"编辑器内工具" → 操作脚本/节点/地图/资源 + UndoRedo  │
│    · 自动拉起 / 关闭 Python 服务                              │
└───────────────────────────┬──────────────────────────────────┘
                            │  本地 HTTP（JSON），HTTPRequest 发起
┌───────────────────────────▼──────────────────────────────────┐
│  [Agent 层] Python LLM 服务（独立进程）                       │
│    · 调用大模型（Tool Calling、思考、上下文缓存）             │
│    · 维护多轮对话状态、路由工具调用                           │
│    · 执行"服务端工具"：文档 prose 补全、代码库检索(RAG)       │
└───────────────────────────────────────────────────────────────┘
```

### 4.2 关键架构决策

- **AR-1**：三层架构；前端用 `HTTPRequest` 调用本地 Python 服务（默认 `127.0.0.1`，端口可配置）。
- **AR-2**：**工具分两类按位置执行**：
  - **编辑器内工具**（操作脚本/节点/地图/资源）：在**前端**执行，因为要操作引擎对象。
  - **服务端工具**（官方文档 prose 补全、工程代码检索等只读重活）：在 **Python 服务**执行；类的**真实签名由前端 ClassDB 提供**（混合接地，见 §5.1）。
- **AR-3**：**改动型工具走「预览—确认」**：模型给出改动 → 前端渲染预览 → 用户确认 → 前端落地并登记 UndoRedo → 回传结果；只读工具直接执行。
- **AR-4**：Agent 循环横跨两进程；对话状态集中服务端，前端每轮只发增量。
- **AR-5**：Python 服务由前端作为**子进程自动启停**，对用户透明。
- **AR-6**：范围含编程辅助与代码检索，Python 层价值凸显（文档 prose、RAG 适合放服务端；类签名取自前端 ClassDB）。

### 4.3 实现路径

- 本期以 **编辑器插件 + Python 服务**形式交付，便于快速迭代、独立发布。
- 工具集与通信协议保持稳定；后续若内置进引擎分支（C++ 模块），交互层不变。

## 5. 能力域与工具集

> 工具按能力域分组、可扩展，**各域同步交付**。LLM 负责"决策/理解"，工具负责"确定性执行"。改动型工具一律先预览后落地。

### 5.1 编程辅助（Programming，支持 GDScript 4.x 与 C#）

| 工具 | 类型 | 功能 |
|------|------|------|
| `read_script` | 只读 | 读取指定/当前脚本内容 |
| `propose_script_edit` | 改动 | 提出脚本修改（生成 diff 供预览确认后写入，带 UndoRedo） |
| `list_project_files` | 只读 | 按过滤条件列出工程文件 |
| `read_class_docs` | 只读（前端 ClassDB + 服务端 prose） | 混合接地：前端取真实签名（含自定义类/GDExtension），服务端补官方 prose 描述，降低幻觉 |
| `search_codebase` | 只读（服务端） | 在工程代码中检索相关片段（RAG，M2） |
| `read_debugger_errors` | 只读 | 读取编辑器/调试器输出的运行时报错，供"修运行时报错"（M2） |
| `propose_tests` | 改动 | 生成单元测试（GUT 等），diff 预览确认后写入（M2） |
| `run_tests` | 执行 | 运行测试并读取结果（M2） |

约束：

- 生成代码须符合目标语言规范——**GDScript 用 4.x 语法**；**C# 用 Godot .NET 现代写法**（如 `[Export]`、`partial class` 等）。
- 自动按文件后缀/脚本类型判断语言（`.gd` → GDScript，`.cs` → C#），并在 prompt 中据此约束。
- **C# 相关工具仅在项目启用 Godot .NET 时暴露**；非 .NET 项目自动隐藏 C# 能力（不报错、不降级），仅提供 GDScript 辅助。
- 修改前先读、改后可撤销；落地前必经 diff 预览确认。

### 5.2 地图建造（Map Building）

| 工具 | 类型 | 功能 |
|------|------|------|
| `fill_rect` | 改动 | 用某瓦片填充实心矩形 |
| `draw_rect_border` | 改动 | 绘制矩形边框（墙/围边） |
| `draw_line` | 改动 | 两点间画线（走廊/路） |
| `set_cells` | 改动 | 批量绘制离散格子 |
| `clear_rect` | 改动 | 擦除矩形区域 |

约束：仅使用上下文中列出的合法瓦片标识 (source_id, atlas 坐标)；先读已用区域避免覆盖；改动以操作清单预览。

### 5.3 场景 / 节点（Scene / Node）

| 工具 | 类型 | 功能 |
|------|------|------|
| `read_scene_tree` | 只读 | 读取当前场景结构 |
| `add_node` | 改动 | 在指定父节点下创建节点 |
| `set_node_property` | 改动 | 设置节点属性 |
| `instance_scene` | 改动 | 实例化预制场景到当前场景 |

约束：改动走 UndoRedo + 预览确认；属性/类型对照 ClassDB 校验。

### 5.4 资源 / 项目（Resource / Project）

| 工具 | 类型 | 功能 |
|------|------|------|
| `create_resource` | 改动 | 创建/保存资源文件 |
| `batch_rename` | 改动 | 批量重命名 |
| `read_project_settings` | 只读 | 读取项目设置 |
| `set_project_setting` | 改动 | 修改项目设置 |

### 5.5 工具架构原则

- **分域、可插拔**：每个能力域是一组工具，可独立增删。
- **三类工具：只读 / 改动 / 执行**：
  - *只读*（读脚本/场景/ClassDB/检索）→ 直接执行。
  - *改动*（写脚本/建节点/绘图/改资源）→ 必经预览确认 + 可撤销（见 §8 NFR-3）。
  - *执行*（`run_tests`、M3 的 AI 试玩/自测）→ 既非只读也非普通改动：需**用户确认 + 超时 + 可取消 + 运行日志 + 沙箱/受控边界**。
- **能力域口径**：§1.4 按能力描述列 5 类（编程/地图/场景/资源/项目）；**工具实现上资源与项目合并为一个 `resource` 域**，故工具域共四类（编程/地图/场景/资源），与里程碑 M1 一致。
- **高层优先**：能用一个高层工具解决的，不让模型逐项输出（省 token、少出错）。
- **位置明确**：每个工具标明在前端还是服务端执行（见 AR-2）。
- **工具增长不改协议**：通信协议与 Agent 循环对工具数量无感。

## 7. 关键业务流程

### 7.1 通用 Agent 流程（以"给玩家加二段跳"为例）

```
1. 用户在面板输入需求并发送
2. 前端收集上下文：选中的玩家节点 + 其脚本(.gd/.cs) + 场景结构，随消息发给服务
3. 服务调用大模型（携带工具定义）
4. 模型可能先调只读工具（read_class_docs 查 CharacterBody2D 真实 API）— 直接执行
5. 模型给出改动型工具调用（propose_script_edit 改玩家脚本）
6. 前端把改动渲染为 diff 预览，等待用户确认 —— 不直接落地
7. 用户点"应用" → 前端写入脚本，登记到同一 UndoRedo 动作
   （用户点"拒绝" → 把"已拒绝"作为结果回传，模型可改方案）
8. 前端回传执行结果
9. 循环 4–8，直至模型给出最终说明
10. 前端提交 UndoRedo（整批可一键撤销），展示改动摘要
```

### 7.2 异常流程

- 无合适上下文（如未选中对象）：提示用户先选中目标。
- 模型产出非法内容（不存在的 API/瓦片、错误属性）：前端/服务校验，作为错误结果回传让模型修正。
- 用户拒绝改动：作为结果回传，模型据反馈调整或终止。
- 服务未就绪/调用失败：前端展示可读错误，不崩溃、不阻塞。

## 8. 非功能需求（Non-Functional Requirements）

| 编号 | 类别 | 需求 |
|------|------|------|
| NFR-1 | 性能 | LLM 调用异步，绝不阻塞编辑器主线程 |
| NFR-2 | 可控性 | 所有改动型操作必须先预览、用户确认后才落地 |
| NFR-3 | 可逆性 | 一轮改动落地后可经**统一撤销入口**一次回退：节点/属性用 Godot 原生 UndoRedo；**文件类改动（脚本写入、资源创建、批量重命名）用快照/事务记录回滚**，统一接入同一撤销栈 |
| NFR-4 | 安全 | 大模型 endpoint/key/模型名仅存本地配置/环境变量，禁止写入场景/脚本/导出包 |
| NFR-5 | 代码安全 | 改动工程文件前先读、预览确认、改后可撤销；不静默覆盖、不破坏无关文件 |
| NFR-6 | 准确性 | 编程：API 文档接地 + 按语言约束（GDScript 4.x / 现代 C#）；地图：瓦片合法性校验；通用：先读后写 |
| NFR-7 | 健壮性 | 任意层失败降级为可读提示，不影响编辑器其余功能 |
| NFR-8 | 成本/上下文 | 上下文按需检索而非全量；启用上下文缓存降本 |
| NFR-9 | 嵌入体验 | 编辑器启动即可用，自动拉起/关闭后台服务，用户零运维 |
| NFR-10 | 可扩展 | 新增能力域/工具不改动核心协议；Agent / Skill / OutputStyle 采用 Claude Code 同构的 markdown/frontmatter 扩展模式，但扩展内容不能授予工具权限、绕过预览确认或修改安全边界 |
| NFR-11 | 可移植 | 前端不依赖特定 OS 特性；服务进程生命周期随编辑器 |
| NFR-12 | 隐私 | 默认本地端点优先；配置**远程端点**前须明确提示“将向该端点发送脚本/场景/错误日志等工程上下文”，用户知情同意；密钥仅本地 |

## 9. 风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| API 幻觉（编造不存在的 Godot API） | 代码跑不起来 | `read_class_docs` 混合接地（前端 ClassDB 真实签名 + 服务端 prose）；落地前可校验 |
| 脚本语言版本/写法（GDScript 出 3.x 旧语法；C# 写法不符 Godot .NET 规范） | 语法/编译错误 | 按文件类型判定语言；system prompt 强约束 4.x / 现代 C#；给少量正例 |
| 模型幻觉（瓦片/属性/坐标非法） | 误绘、报错 | 上下文提供合法集合；执行前校验；非法回传让其修正 |
| 工程规模大，上下文塞不下 | 慢、贵、漏信息 | 按需检索（RAG）+ 只给相关片段，不全量 |
| 破坏用户既有代码/场景/地图 | 损坏成果 | 先读后写、预览确认、避让既有内容、一切可撤销 |
| 阻塞编辑器 | 卡顿 | 全异步、信号驱动 |
| 密钥泄露 | 安全事故 | 仅环境变量；不进版本库/导出包 |
| 隐私 / 数据出域（远程端点） | 工程脚本/场景/错误日志被发往远程 LLM | 默认本地端点；远程端点首次使用明确提示并需用户同意（NFR-12）；可配置不发送哪些上下文 |
| 不同厂商 LLM 端点 API 格式不一（OpenAI 兼容 / Anthropic 等） | 配置后调用失败 | 设计阶段确定适配方式（约定兼容某类协议或加端点适配层）；配置/调用错误给可读提示 |
| C# 仅 .NET 项目可用 | 部分用户无 C# 能力 | 运行时检测是否 .NET 项目，自动隐藏 C# 工具，仅保留 GDScript |
| Python 服务运维负担 | 体验差 | 编辑器自动拉起/关闭子进程 |
| 跨进程 Agent 循环复杂度 | 稳定性 | 对话状态集中服务端；前端只发增量；明确错误降级 |
| Agent/Skill/OutputStyle 扩展被误用为授权来源 | 安全/越权 | 扩展内容只作为提示词/配置资产；实际能力仍由类型化工具、权限闸、前端预览确认决定；项目级扩展默认需显式启用 |
| 范围大、各域同步交付 | 交付/质量风险 | M0 先打通骨架与预览确认机制，M1 各域核心工具一起实现，避免一次性堆砌 |

## 10. 里程碑（建议）

> 各能力域同步交付，但仍以"先骨架、后铺开、再增强"的顺序推进，控制交付风险。

| 阶段 | 内容 | 验收标准 |
|------|------|----------|

## 11. 决策记录与待细化项（Decisions & Open Items）

### 11.1 已确定决策

| # | 决策 |
|---|------|
| D1 | 各能力域**同步交付**（先骨架、后铺开） |
| D2 | 编程辅助**同时支持 GDScript 4.x 与 C#**；C# 仅在 .NET 项目可用，非 .NET 自动隐藏、不降级 |
| D3 | 所有改动**先预览（diff/操作清单）、确认后落地**，落地后仍可撤销 |
| D4 | 大模型 **endpoint、key、模型名由用户本地配置**，支持任意可配置端点；**默认本地优先，使用远程端点会发送工程上下文，须向用户明确提示并征得同意**（见 NFR-12、风险表） |
| D5 | 预览粒度：**M1 整批应用/拒绝，M2 增加逐条** |
| D6 | Agent / Skill / OutputStyle 扩展采用 Claude Code 同构 markdown/frontmatter 模式；扩展只影响提示词、选择和表达方式，不能提权 |

### 11.2 待设计阶段细化

1. **端点适配**：用户可填入不同厂商端点（OpenAI 兼容 / Anthropic 等），其请求/响应与工具调用格式不一——设计阶段需定下"约定兼容某类协议"还是"做端点适配层"。
2. **代码库 RAG**：纳入 M2；索引何时构建（首次打开工程 / 手动触发 / 增量更新）待定。
3. **C# 检测方式**：如何在运行时判定项目是否启用 Godot .NET，并据此暴露/隐藏 C# 工具。
4. **通信协议与预览数据结构**：前后端消息体、工具调用与"可预览改动"的具体字段格式，可在设计阶段补一份接口规格。
